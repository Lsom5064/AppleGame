# Team Mode Manual Test Checklist

이 문서는 `goals/current-goal.md`의 남은 수동 검증 항목을 실제로 점검하기 위한 체크리스트다.

## 준비

- 기기 또는 브라우저 인스턴스 4개를 준비한다.
- 권장 조합:
  - 데스크톱 Chrome
  - 데스크톱 Safari 또는 Firefox
  - 모바일 Safari 또는 Chrome
  - 시크릿 창 또는 다른 프로필
- Firebase 연결이 실제로 활성화된 환경에서 테스트한다.
- 테스트 닉네임 예시:
  - `Host`
  - `GuestA`
  - `GuestB`
  - `GuestC`

## 공통 확인 항목

- 방 생성, 참가, 나가기, 재접속 중 콘솔 오류가 없는지 확인한다.
- 채팅, 팀 배정, 점수판, 포인터, 라운드 상태가 모든 화면에서 같은 의미로 보이는지 확인한다.
- 새로고침 후 방 상태가 복구되더라도 중복 플레이어가 생기지 않는지 확인한다.

## 시나리오 1: 개인전 회귀

1. `Host`가 개인전 방을 만든다.
2. `GuestA`가 참가한다.
3. `1판`, `3판`, `5판` 각각 시작해본다.
4. `sum`, `best` 리더보드 기준을 각각 확인한다.

기대 결과:
- 기존 개인전 흐름이 깨지지 않는다.
- 점수 제출, 결과 화면, 재시작, 나가기 모두 정상 동작한다.

## 시나리오 2: 개별 화면 팀전

1. `Host`가 팀전 `개별 화면`으로 바꾼다.
2. 팀 수를 `2`, `3`, `4`, `6`으로 각각 바꿔본다.
3. 랜덤 팀 배정 버튼을 누른다.
4. 수동 배정으로 팀을 다시 바꾼다.
5. 바로 게임을 시작한다.

기대 결과:
- 팀 수 변경 직후에도 잘못된 팀 배정이 남지 않는다.
- 수동으로 옮긴 팀 배정은 시작 후 그대로 유지된다.
- 결과 화면에서 팀 점수와 팀 내부 개인 점수가 논리적으로 맞는다.

## 시나리오 3: 단일 화면 팀전 기본

1. `Host`, `GuestA`를 `1팀`, `GuestB`, `GuestC`를 `2팀`으로 둔다.
2. 팀전 `단일 화면`으로 시작한다.
3. `Host`와 `GuestA`가 같은 팀 보드를 보고 있는지 확인한다.
4. `GuestB`, `GuestC`는 자기 팀 보드 기준으로 동작하는지 확인한다.

기대 결과:
- 같은 팀은 같은 보드 진행도를 본다.
- 다른 팀의 진행도는 섞이지 않는다.

## 시나리오 4: 포인터 동기화

1. 단일 화면 팀전 중 `Host`가 드래그를 시작한다.
2. `GuestA` 화면에서 `Host`의 포인터가 보이는지 확인한다.
3. `Host`가 포인터를 빠르게 움직인다.
4. `Host` 탭을 백그라운드로 보내거나 새로고침한다.

기대 결과:
- 팀원 포인터가 실시간으로 따라온다.
- 끊긴 포인터는 잠시 후 사라진다.
- 재접속 후 같은 플레이어로 복구되고, 중복 포인터가 남지 않는다.

## 시나리오 5: 사과 제거 동기화

1. 단일 화면 팀전 중 `Host`가 합 10 선택을 만든다.
2. `GuestA` 화면에서 같은 사과가 즉시 떨어지고 제거되는지 본다.
3. 제거 직후 점수가 두 화면에서 일치하는지 본다.

기대 결과:
- 한 팀원이 제거한 사과는 같은 팀 화면에 즉시 반영된다.
- 점수와 제거 상태가 팀 내에서 동일하다.

## 시나리오 6: 중복 제거 충돌

1. `Host`와 `GuestA`가 같은 팀에서 같은 사과 묶음을 거의 동시에 드래그한다.
2. 같은 팀의 다른 기기에서도 결과를 본다.

기대 결과:
- 점수가 한 번만 반영된다.
- 같은 사과가 두 번 제거되지 않는다.
- 보드 상태가 깨지지 않는다.

## 시나리오 7: 재접속 복구

1. 단일 화면 팀전 도중 `GuestA`를 새로고침한다.
2. 개별 화면 팀전 도중 `GuestB`를 새로고침한다.
3. 로비, 라운드 대기, 결과 화면에서도 각각 새로고침해본다.

기대 결과:
- 같은 `playerId`로 복구된다.
- 닉네임, 팀, 점수, 제출 상태가 유지된다.
- 단일 화면 팀전에서는 팀 보드 제거 상태도 유지된다.

## 시나리오 8: 오프라인/복귀

1. 플레이 중 `GuestB` 네트워크를 잠시 끊는다.
2. 남은 인원만으로 라운드를 끝낸다.
3. `GuestB`를 다시 온라인으로 복귀시킨다.

기대 결과:
- 오프라인 플레이어가 라운드 종료와 다음 라운드 투표를 영구적으로 막지 않는다.
- 복귀 후 연결 상태가 다시 보인다.
- shared mode에서는 복귀 후 팀 보드 상태가 현재 진행도 기준으로 복구된다.

## 시나리오 9: 채팅 동시성

1. 로비에서 동시에 여러 명이 채팅을 보낸다.
2. 라운드 대기 중 채팅을 보낸다.
3. 결과 화면에서 채팅을 보낸다.
4. shared mode에서 포인터 이동/사과 제거와 동시에 채팅을 보낸다.

기대 결과:
- 메시지가 유실되거나 순서가 이상하게 깨지지 않는다.
- 채팅이 게임 상태 동기화를 망치지 않는다.

## 시나리오 10: goal 20 근접 테스트

1. 단일 화면 팀전 2팀 4인 방을 만든다.
2. `Host`, `GuestA`는 1팀, `GuestB`, `GuestC`는 2팀으로 둔다.
3. 같은 순간:
   - `Host`, `GuestA`는 같은 사과 묶음을 클릭한다.
   - `GuestB`, `GuestC`도 자기 팀 보드에서 같은 묶음을 클릭한다.
   - `GuestA`는 바로 새로고침한다.
   - `GuestB`는 잠시 네트워크를 끊는다.
   - `Host`는 채팅을 보낸다.
   - 라운드 종료 직전까지 시간을 보낸다.

기대 결과:
- 각 팀의 점수는 한 번씩만 반영된다.
- 제거된 사과 상태가 팀 내에서 일치한다.
- 포인터는 끊긴 플레이어 기준으로 정리된다.
- 채팅은 남고, 라운드 종료 후 결과가 논리적으로 맞는다.
- 재접속 플레이어는 중복 참가자 없이 복구된다.
