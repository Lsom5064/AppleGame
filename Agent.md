# Team Battle Expansion Notes

## 가능성 판단

이 요구는 현재 코드베이스와 Firebase Realtime Database 구조 위에서도 구현 가능합니다. 다만 난이도는 크게 두 단계로 나뉩니다.

1. `개별 화면 팀전`
- 구현 가능성이 높고, 현재 구조를 확장하는 방향으로 처리할 수 있습니다.
- 현재도 플레이어별 점수, 라운드, 리더보드, 방 설정이 이미 방 상태에 있으므로 `팀`, `팀 배정`, `팀 점수 집계`를 추가하면 됩니다.

2. `단일 화면 팀전`
- 구현 가능하지만, 현재 구조의 핵심 가정을 바꿔야 합니다.
- 지금 게임 로직은 각 클라이언트가 `room seed + round index`로 보드를 로컬 생성하고, 최종 점수만 동기화합니다.
- 단일 화면 팀전은 이 방식으로는 부족합니다. 팀원 간에 같은 보드를 실시간으로 공유해야 하므로, `사과 제거 상태`, `팀 공용 점수`, `팀원 포인터 위치`, `팀별 현재 보드 상태`를 실시간 동기화하는 별도 상태 모델이 필요합니다.
- 즉, 단순 옵션 추가가 아니라 `인게임 상태 동기화 모델`을 한 단계 더 세분화해야 합니다.

결론:
- 요구사항 전체는 가능
- `개별 화면 팀전`은 중간 난이도
- `단일 화면 팀전`은 대규모 구조 변경

## 요구사항 정리

### 공통 팀전 요구

1. 대기실에서 게임 모드를 `개인전 / 팀전`으로 변경할 수 있어야 함
2. 팀전 모드에서 팀 수를 선택할 수 있어야 함
3. 팀전 모드에서 랜덤 팀 배정 버튼이 있어야 함
4. 방장이 수동으로 팀을 직접 조정할 수 있어야 함

### 팀전 하위 모드

1. `개별 화면 팀전`
- 각 플레이어는 기존처럼 각자 자기 화면에서 플레이
- 라운드 종료 후 플레이어별 점수를 팀별로 합산
- 최종 리더보드는 `개인 랭킹`이 아니라 `팀 랭킹` 중심으로 표시

2. `단일 화면 팀전`
- 한 팀이 하나의 공용 게임판을 공유
- 같은 팀 플레이어는 동일한 사과 보드를 봄
- 팀원의 마우스 포인터가 서로에게 보여야 함
- 팀원이 없앤 사과가 다른 팀원 화면에도 즉시 반영돼야 함
- 팀 공용 점수와 공용 진행 상태가 필요함

## 현재 구조에서 필요한 모델 변경

### 방 설정 확장

현재 `RoomState.settings`는 다음 값만 가집니다.
- `roundCount`
- `leaderboardMode`
- `roundDurationSec`

여기에 아래 설정이 추가되어야 합니다.
- `gameMode`: `solo | team`
- `teamMode`: `individual | shared`
- `teamCount`

### 방 상태 확장

현재 `RoomState`에는 팀 정보가 없습니다. 아래 필드가 필요합니다.
- `teams`: 팀 메타데이터 배열 또는 맵
- `playerTeams`: `playerId -> teamId`

단일 화면 팀전까지 지원하려면 추가로 필요합니다.
- `teamBoards` 또는 `sharedBoardStates`
- `teamPointers`
- `teamRoundScores`
- `sharedBoardOperations` 또는 `removedAppleIds`

### 2단계: 개별 화면 팀전

수정 대상:

### 3단계: 단일 화면 팀전용 인게임 상태 모델

이 단계부터는 현재 게임 상태 구조를 더 크게 바꿉니다.

수정 대상:
- `src/types.ts`
- `src/services/realtimeService.ts`
- `src/utils/gameBoard.ts`
- `src/components/GameScreen.tsx`
- `src/components/GameBoard.tsx`

작업:
- 현재 로컬 전용 보드 상태를 팀 공유 상태로 분리
- 팀 단위 보드 상태를 Firebase에 저장
- 사과 제거 결과를 `팀 공용 연산`으로 기록
- 클라이언트는 공용 보드 상태를 구독

핵심 결정:
- 단순 전체 보드 스냅샷 저장
- 또는 사과 제거 이벤트 로그 저장

현 구조에서는 `이벤트 로그 + 최종 상태 반영` 쪽이 더 안전합니다.

### 4단계: 팀 포인터 동기화

수정 대상:

- `src/services/realtimeService.ts`

작업:
- 팀원 포인터 좌표를 실시간 저장
- 같은 팀 플레이어만 포인터를 구독
- 오래된 포인터는 timestamp 기준으로 정리

### 5단계: 보안/정합성/UX 보강

작업:
- Firebase rules 확장
- 팀원 수동 변경 중 충돌 방지
- 팀 공용 보드 연산 중복 적용 방지
- 재접속 시 팀/공용 보드 복구
- 단일 화면 팀전의 제출/종료 규칙 정리

## 기술적 주의점

1. 현재는 `점수 제출형` 구조다
- 플레이 중 세부 상태를 거의 공유하지 않음
- 단일 화면 팀전은 이 가정을 깨는 첫 기능임

2. Firebase RTDB로도 가능하지만, 완전한 실시간 협업 보드라면 구조를 신중히 잡아야 함
- 포인터 동기화는 가능
- 사과 제거 즉시 반영도 가능
- 다만 동시에 같은 사과를 제거하려는 충돌은 transaction 또는 operation id로 막아야 함

3. 성능과 난이도 기준으로는 순차 구현이 맞음
- 먼저 `개별 화면 팀전`
- 그 다음 `단일 화면 팀전`

## 권장 구현 순서

1. 로비에서 팀전 설정/팀 배정
2. 개별 화면 팀전 리더보드
3. 단일 화면 팀전 공용 보드 상태
4. 팀 포인터 동기화
5. 보안/정합성 보강

## Goal 문구

아래 문구를 그대로 목표 문구 초안으로 사용할 수 있습니다.

`현재 Apple Sum 10 Multiplayer 프로젝트에 팀전 모드를 추가한다. 대기실에서 개인전/팀전을 전환할 수 있어야 하고, 팀전에서는 팀 수 설정, 랜덤 팀 배정, 방장의 수동 팀 배정이 가능해야 한다. 팀전은 개별 화면 모드와 단일 화면 모드를 모두 지원해야 한다. 개별 화면 모드는 각자 플레이 후 팀별 점수를 합산하는 방식이어야 한다. 단일 화면 모드는 팀이 하나의 공용 보드를 공유하며, 팀원의 포인터 이동과 사과 제거 결과가 실시간으로 서로의 화면에 반영되어야 한다. Firebase Realtime Database 기반 멀티플레이 구조와 기존 라운드/채팅/재접속 흐름을 유지하면서 단계적으로 구현한다.`
